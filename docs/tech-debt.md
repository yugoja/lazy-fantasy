# Tech Debt — Local / Prod Parity

Living doc. Items are scoped to closing the gap between local dev and production so that "works locally" reliably means "works in prod."

Status legend: `[ ]` not started · `[~]` in progress · `[x]` done

---

## High impact

### 1. `[x]` Switch local DB from SQLite to Postgres
**Gap.** `backend/app/database.py` branches engine config on `sqlite:` vs Postgres (`database.py:7-15`). Local devs run SQLite (`DATABASE_URL=sqlite:///./fantasy_cricket.db` per `.env.example`); prod runs Postgres on the droplet. Dialect differences silently mask bugs: case sensitivity, FK enforcement, JSON types, transaction isolation, `ORDER BY NULL` behavior, datetime/timezone handling.
**Done.** `backend/.env.example` now defaults to the Docker Postgres URL. `docker-compose.yml` (#5) provides `postgres:16` matching prod major version. `backend/database.py` SQLite branch remains but is dead code — tracked for deletion in #13.

### 2. `[x]` No migration runner / version tracking — **Alembic live on prod (`a1c99c58f11a`)**
**Gap.** `backend/migrations/pg/` was 7 loose `.sql` files with no `schema_migrations` table.
**Done.** Alembic added to `backend/requirements.txt`. `backend/alembic/` scaffolded; `env.py` reads `settings.DATABASE_URL` + `Base.metadata`. Baseline revision `a1c99c58f11a` autogenerated against a fresh Postgres and verified to create all 14 tables with every historical-migration column intact (`picks_window`, `sport`, `is_archived`, `display_name`, `cricapi_player_id`, `external_match_id`+sync fields). `backend/migrations/README.md` documents the workflow; `backend/Makefile` provides `make migrate`, `make migration NAME=…`, `make migrate-down`.
**Done.** `alembic stamp head` run on prod 2026-05-16; `SELECT * FROM alembic_version` returns `a1c99c58f11a`. Future schema changes go through `alembic revision --autogenerate` + `make migrate`.

### 3. `[x]` High-quality seed data for end-to-end testing
**Gap.** `backend/scripts/seed_ipl2026_fixtures.py` exists but only covers fixtures. No realistic users, leagues, predictions, results, or dugout events. End-to-end smoke testing locally is manual click-throughs from an empty DB.
**Done.** `backend/scripts/seed_dev.py`: 10 fake users (`seed_user_0..9` / `devpass`), 3 leagues with overlapping membership, 5 completed past matches with scored predictions, 3 upcoming matches with partial predictions. Idempotent — safe to re-run. Wired into `make reset` (root and `backend/Makefile`) and exposed as standalone `make seed`.

### 4. `[ ]` Tests likely run on SQLite
**Gap.** `backend/tests/conftest.py` exists; default test DB is almost certainly SQLite for speed. Means scoring/aggregation tests can pass locally and fail on Postgres prod (transaction isolation, real FK enforcement, `ORDER BY` determinism).
**Fix.** Move test DB to Postgres via `testcontainers-python` (spins ephemeral PG per test session) or a session-scoped fixture that targets a local Postgres `lazyfantasy_test` DB. Blocked by #1.

### 5. `[x]` No Docker / docker-compose for local stack
**Gap.** No `Dockerfile`, no `docker-compose.yml`. Onboarding requires: install Postgres locally, configure VAPID keys, set up Resend, pin Python version, etc. Postgres version drift between dev machines and prod (Render uses a specific PG version on the droplet) is invisible.
**Done.** `docker-compose.yml` has `postgres:16` + `backend` (Python 3.12-slim) + `frontend` (Node 20-alpine) services. Backend entrypoint runs `alembic upgrade head` before starting uvicorn. `Makefile` at repo root: `make up`, `make down`, `make logs`, `make reset`. `backend/docker-entrypoint.sh` runs migrations on every container start. Hot-reload via bind mounts on both services. Closes #1 (SQLite branch now dead code) and #12.

## Medium impact

### 6. `[ ]` Anonymized prod-dump → local sync script
**Gap.** CLAUDE.md TODO mentions `db-sync.sh` for `pg_dump` prod → restore locally, but no anonymization plan. Real prod data contains: user emails, Google IDs, display names, push subscription endpoints. Restoring as-is locally is a privacy hazard *and* will fire real push notifications to real users if the scheduler runs against the restored DB.
**Fix.** Write `scripts/db-sync.sh` that: (a) `pg_dump` prod, (b) restore to local `lazyfantasy_dev`, (c) run an anonymization SQL: scrub `users.email` → `user{id}@example.com`, null `users.google_id`, randomize `users.display_name`, `DELETE FROM push_subscriptions`, `DELETE FROM reminder_log`. Optionally rotate the local JWT secret too. Document run cadence (weekly? on-demand?).

### 7. `[ ]` Background scheduler hard to test locally
**Gap.** `app/services/scheduler.py:_send_match_reminders()` runs every 5 min via APScheduler in-process, looking for matches starting 55–65 min from now. Locally, you almost never have a match in that window — reminders silently work in prod and break locally without you ever knowing.
**Fix.** Add `backend/scripts/fire_reminder_for_match.py <match_id>` that invokes the scheduler's send logic directly with widened time bounds. Manual trigger for local testing. Bonus: a `/admin/debug/fire_reminder/{match_id}` endpoint gated to dev mode.

### 8. `[ ]` Backend ↔ frontend type drift
**Gap.** `frontend/src/types/index.ts` hand-mirrors backend Pydantic schemas. Every prediction/match shape change requires manual updates on both sides; drift is silent until a runtime error in the browser.
**Fix.** Add `openapi-typescript` to frontend; FastAPI emits `/openapi.json` at startup; generate `src/types/api.ts` as part of `npm run dev` or a `npm run gen:types` script. Existing hand-written types in `src/types/index.ts` get replaced or augmented. Especially valuable for the upcoming multi-sport refactor (polymorphic payloads will have non-trivial union types).

### 9. `[ ]` No CI workflows
**Gap.** No `.github/workflows/` directory. Tests and lint only run when devs remember to run them locally — and locally means SQLite (see #4), so passing locally isn't proof of prod readiness.
**Fix.** Minimal GitHub Actions pipeline on PR + push to `main`/`staging`: boot Postgres service, install backend deps, run `alembic upgrade head`, run pytest. In parallel: install frontend deps, run `npm run test:run`, run `npm run lint`, run `tsc --noEmit`. Blocked by #2 for the migration step to be meaningful.

## Lower impact

### 10. `[ ]` Disabled features silently no-op locally
**Gap.** `.env.example` lists optional features with blank defaults: `RESEND_API_KEY`, `VAPID_PRIVATE_KEY`, `SENTRY_DSN`, etc. Code paths degrade silently when these are missing — devs don't realize email/push/error-tracking is broken in their local environment, leading to "works on my machine" that ships broken behavior.
**Fix.** At app startup, log a `WARNING` for each disabled feature listing the env var to set. Add `scripts/dev-setup.sh` that generates dev VAPID keys and prints what else to configure for full local fidelity.

### 11. `[ ]` Local HTTPS for PWA / push notification testing
**Gap.** Push notifications and `serviceWorker` registration only work on HTTPS or `localhost`. `localhost` works for desktop Chrome but breaks when testing on a phone over LAN (HTTP to LAN IP is rejected by the browser).
**Fix.** Document `ngrok` usage in `frontend/README.md` for phone testing, or add a `mkcert` step (`make https`) that issues a local CA cert + serves frontend over HTTPS.

### 12. `[x]` No `make reset` / data-reset command
**Gap.** Wiping local DB for clean testing is manual: drop DB → recreate → run all `.sql` files in some order → seed fixtures.
**Done.** `make reset` at repo root: wipes DB volume, rebuilds containers, waits for migrations. Still needs `seed_dev.py` (#3) to be called here for full usefulness.

### 13. `[ ]` SQLite/Postgres branching in `database.py`
**Gap.** `database.py:7-15` carries an `if settings.DATABASE_URL.startswith("sqlite"):` branch. Once everyone's on Postgres locally (#1), this branch is dead code that signals "we support both" when we shouldn't.
**Fix.** Delete the branch; assume Postgres. Blocked by #1.
